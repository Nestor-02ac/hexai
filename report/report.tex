\begin{abstract}
This repository studies Monte Carlo Tree Search across two connection games,
Hex and Y, inside one shared implementation framework. The project has three
active branches: a classical Hex baseline reproducing the UCT+RAVE setting of
\emph{Monte-Carlo Hex}, a learned Hex agent based on Gumbel AlphaZero, and a
new classical implementation for Y. The main engineering goal is consistency:
the classical branches expose the same Python and Cython APIs, use incremental
connectivity tracking, and can be compared without conflating algorithmic
differences with implementation asymmetries. The main research takeaway is
twofold. First, the classical Hex results preserve the qualitative behavior
reported in the literature even with simpler rollouts. Second, the new Y
implementation now matches the Hex branch in logic and speed infrastructure.
Its full-size experiments show the expected simulation-scaling curve, a strong
preference for $C_{\mathrm{uct}}=0$, and a clear benefit from
connectivity-aware rollouts, while also making clear that Y still lacks an
analogue of Hex bridge knowledge with equally strong structural impact.
\end{abstract}

\section{Introduction}

Monte Carlo Tree Search (MCTS) is a simulation-based search procedure designed
for sequential decision problems with large game trees and difficult static
evaluation functions. Starting from a root state, it repeatedly alternates
between selection, expansion, simulation, and backpropagation. Rather than
distributing work uniformly, it allocates simulations toward the lines of play
that currently appear most promising. This makes it particularly well suited to
connection games, where local moves may have long-range strategic
consequences and exhaustive search becomes impractical very quickly.

This project began as a study of Hex and then expanded into a broader
connection-game repository. The current codebase contains three active modules.
\texttt{HexClassic/} implements a classical UCT+RAVE baseline inspired by
Cazenave and Saffidine's \emph{Monte-Carlo Hex} \cite{cazenave2010montecarlohex}.
\texttt{HexGumbel/} replaces rollout evaluation with a learned policy-value
model and follows the Gumbel AlphaZero search scheme
\cite{danihelka2022gumbel}. \texttt{YClassic/} extends the classical branch to
the game of Y while preserving the same implementation standards as the Hex
baseline.

The report therefore has a double purpose. It documents the classical and
learned Hex systems, but it also explains how the Y branch was upgraded from a
draft into a fully usable experimental module. The central idea is that the
repository should read as one coherent study, not as three unrelated projects.
Shared engineering decisions are described once, and game-specific differences
are explained only where the geometry, rules, or available heuristics actually
diverge.

\section{Literature Review and State of the Art}

The study of Hex-playing algorithms dates back to the early history of
game-playing artificial intelligence. One of the foundational discussions of
computer approaches to board games appears in Shannon's classic paper on
computer chess \cite{shannon1950chess}, which helped establish the broader idea
that strategic play can be approached through search guided by approximate
evaluation rather than exhaustive enumeration.

Early approaches to game playing relied on minimax search combined with
alpha-beta pruning, which systematically explores the game tree while removing
branches that cannot influence the final decision. While effective in games
such as chess, these methods are less natural for Hex because of its large
branching factor and the difficulty of designing static evaluation functions
that capture global connectivity well.

Modern approaches to Hex have been largely shaped by Monte Carlo Tree Search.
Cazenave and Saffidine \cite{cazenave2010montecarlohex} introduced a framework
combining UCT with the RAVE heuristic, significantly improving the efficiency
of search by sharing information between simulations. This paper is the direct
methodological baseline for the classical branch of this repository.

Building on these ideas, Arneson, Hayward, and Henderson developed MoHex
\cite{arneson2010mctshex}, a strong Hex-playing program based on MCTS that
achieved top-level performance in computer Hex competitions. Subsequent work
further strengthened this direction by incorporating learning components. In
particular, Gao, M\"uller, and Hayward proposed a three-head neural
architecture for MCTS in Hex \cite{gao2018threehead}, improving both move
selection and position evaluation by enriching the information returned by the
network.

More recent research has explored deeper integration of neural networks and
reinforcement learning. Takada et al. introduced the DeepEzo line of work
\cite{takada2020reinforcement}, which learns value and policy functions for Hex
through self-play and search. Cazenave's Polygames framework
\cite{cazenave2020polygames} then showed more broadly that zero-learning style
systems can be transferred across several board games within one shared
framework. In parallel, AlphaGo Zero made the policy-value paradigm central to
modern game AI \cite{silver2017mastering}, and Gumbel AlphaZero refined the
root-search procedure itself \cite{danihelka2022gumbel}. These developments are
the natural background for the learned Hex branch in this repository.

The Y extension occupies a slightly different position in this landscape. There
is a substantial methodological literature on MCTS and a substantial applied
literature on Hex, but far less standardized experimental literature on modern
Y engines. For that reason, the Y branch in this repository should be read less
as a reproduction of one canonical prior benchmark and more as a transfer of
the connection-game search toolbox developed for Hex to a more general
three-sided setting. Its rule model is grounded in standard descriptions of Y
itself \cite{boardspace_y,gambiter_y}, while its algorithmic foundations come
primarily from the Hex and general MCTS literature above.

\section{Games and Scope}

\subsection{Hex}

Hex is played on an $N \times N$ rhombus of hexagonal cells. One player tries
to connect the top and bottom borders, while the other tries to connect the
left and right borders. Players alternate placing stones on empty cells, and
stones are never moved or removed. The game has no draws, which is one reason
it is a natural domain for Monte Carlo evaluation: every complete simulation
terminates with a unique winner.

From a search perspective, Hex is difficult because its value depends on global
connectivity rather than on local material balance. A move that looks modest in
isolation may become decisive once it links two larger structures elsewhere on
the board. This is precisely the kind of setting in which simulation-based
search became attractive.

\subsection{Y}

Y belongs to the same family of connection games but changes the geometry and
the win condition. The version used in this repository is the standard
triangular hex-grid form of Y described by Boardspace and Gambiter
\cite{boardspace_y,gambiter_y}. A player wins by forming one connected
component touching all three sides of the triangle. Corner cells count as
belonging to both adjacent sides, and a full board has exactly one winner
\cite{boardspace_y,gambiter_y}. Standard descriptions also note an alternative
``official'' board with three five-neighbor points, but that is not the model
used here. We adopt the regular triangular grid because it provides the cleanest
comparison with Hex and the most direct extension of the existing code.

\section{Methods}

\subsection{Classical MCTS with UCT and RAVE}

The classical baseline follows the UCT+RAVE structure introduced in
\emph{Monte-Carlo Hex} \cite{cazenave2010montecarlohex}. During selection, each
action is scored by the empirical action value augmented with the usual UCT
exploration bonus:
\[
\mathrm{UCT}(s,a)=Q(s,a)+C_{\mathrm{uct}}
\sqrt{\frac{\log N(s)}{N(s,a)}}.
\]

RAVE, also known as AMAF, supplements this estimate by reusing information from
later simulation moves. Informally, if a move proves good in a playout, RAVE
treats it as evidence that the move might also have been good if played
earlier. This is especially useful at the start of search, when ordinary visit
counts are still sparse. In our implementation, the Monte Carlo estimate and
the RAVE estimate are blended with the same bias-controlled interpolation used
in \cite{cazenave2010montecarlohex}.

The interesting modeling choice lies in the rollout policy. In Hex, we keep a
lightweight bridge-defense heuristic rather than the stronger template stack
used in the original paper. That makes the rollout policy weaker, but it also
makes the behavior of the UCT+RAVE core easier to isolate. In Y, there is no
direct bridge analogue, so the playout policy is based on local connectivity
criteria instead. In both cases the goal is the same: keep the search logic
fixed, then ask what game-specific structural knowledge matters.

\subsection{Neural-Network-Guided MCTS}

The learned branch replaces stochastic rollout evaluation with a neural
policy-value model. At interior tree nodes, search follows the standard PUCT
rule
\[
Q(s,a)+c_{\mathrm{puct}}\,P(s,a)\frac{\sqrt{N(s)}}{1+N(s,a)},
\]
where $P(s,a)$ is the prior probability predicted by the network. The network
therefore affects the search in two ways: it directs exploration toward moves
that look plausible and it supplies a value estimate in place of full
rollouts.

At the root, the project follows the Gumbel AlphaZero policy-improvement
mechanism of \cite{danihelka2022gumbel}. Candidate moves are perturbed with
Gumbel noise, filtered, and then refined through sequential halving. Relative
to a pure AlphaZero-style training loop \cite{silver2017mastering}, the present
implementation adds supervised pretraining from classical Hex labels before
self-play begins. This is not just a convenience. Given the available compute
budget, the warm start substantially stabilizes early training and avoids the
worst part of the cold-start regime.

\section{Engineering Architecture}

\subsection{Shared Implementation Strategy}

One of the most important design choices in the repository is that the two
classical modules follow the same implementation pattern. Each of them exposes a
pure Python reference path and a faster Cython backend with the same public
API. The Python version keeps the code easy to inspect and test. The Cython
version makes large experimental sweeps practical.

The central low-level issue in both Hex and Y is connectivity maintenance. The
code therefore tracks components incrementally with Union-Find structures rather
than recomputing graph searches after every move. In Hex, components connect to
two virtual borders. In Y, the equivalent information is encoded as a three-bit
side mask. This means that the underlying pattern is shared even though the
rule condition changes.

The optimized backends also follow the same philosophy. Board state is stored
in typed arrays, board copies use \texttt{memcpy}-style cloning, hot-loop MCTS
state is flattened into arrays where possible, and experiment workers are
seeded explicitly so multiprocessing remains reproducible. The point is not
that Hex and Y somehow require different optimization stories. Once the shared
pattern was established for Hex, the Y extension could be upgraded without
becoming a one-off implementation.

\subsection{Module Structure}

At the repository level, the classical and learned code paths remain clearly
separated. The Hex baseline module contains board logic, the classical MCTS
implementation, Cython board and tree-search backends, experiment runners, and
saved JSON results. The learned Hex branch adds configuration, network
definitions, self-play generation, training, and evaluation utilities around a
neural MCTS core. The Y branch mirrors the classical Hex layout with board
logic, Python MCTS, Cython board and MCTS backends, an experiment runner, an
interactive play script, and regression tests.

This organization matters because it makes the Y work legible as an extension
of the existing architecture rather than as an isolated fork. The Y code is not
trying to invent a new interface; it is deliberately conforming to the one
already used by the Hex baseline.

\subsection{Y as an Extension of the Hex Baseline}

The Y module began as a draft that still lacked the maturity of
\texttt{HexClassic/}. The extension work therefore had three objectives:
correctness of the game logic, parity with the Hex architecture, and practical
speed. The resulting board implementation uses the regular triangular geometry,
which contains
\[
n=\frac{s(s+1)}{2}
\]
cells for side length $s$. Each stone belongs to zero, one, or two sides of the
triangle depending on whether it is interior, on an edge, or in a corner.

For each player, every connected component stores a three-bit mask,
\[
\texttt{LEFT}=001,\qquad
\texttt{RIGHT}=010,\qquad
\texttt{BOTTOM}=100.
\]
When a move is played, it merges with neighboring friendly components and the
masks are OR-combined. A win occurs as soon as one merged component reaches
\texttt{111}. This makes Y win detection constant-time after each move, exactly
as the Hex implementation uses incremental connectivity instead of repeated
graph traversal.

The search logic also follows the Hex baseline closely. There is a single
UCT+RAVE loop, standard AMAF-style updates, and full-board playouts. The one
major game-specific difference is the rollout policy. Hex benefits from a
simple but meaningful bridge-defense heuristic. Y has no equally direct local
analogue, so its stronger rollout policy instead prefers moves that merge
friendly components, increase the number of touched sides, or contest
structurally important regions around opposing stones.

Correctness was checked in three complementary ways. Small full boards were
enumerated to confirm that every complete coloring has exactly one winner.
Random game states were compared against a naive connectivity evaluator.
Finally, Python and Cython implementations were compared on identical move
sequences to ensure state-by-state agreement. The result is not merely a faster
draft; it is a module whose logic is now trustworthy enough to support larger
experiments.

On the performance side, the gain from the fast backend is substantial. On an
empty side-length-7 board with 2{,}000 simulations, the Python path takes
0.298\,s while the Cython path takes 0.0135\,s, a 22.1$\times$ speedup. More
importantly for practical experimentation, the full-size matched benchmark is
not dramatically more expensive than Hex. On side length 11 with the Cython
backend and the same 2{,}000-simulation root search, Hex takes 0.0276\,s and Y
takes 0.0377\,s. The Y extension is therefore only about 1.37$\times$ slower in
the non-toy setting.

\section{Discussion}

Taken together, the three modules show three complementary points about MCTS in
connection games.

First, classical search remains a strong and interpretable baseline. The
\texttt{HexClassic/} results confirm that even a deliberately simplified
rollout policy is enough to preserve the main UCT+RAVE trends from the
literature. Second, the learned branch demonstrates that search quality is not
only a matter of simulation count. Once the prior and value function are good
enough, a neural agent can outperform a far more heavily simulated classical
opponent. Third, extending an existing search system to a new game requires far
more than changing the board geometry. The Y extension worked because the
shared architecture was strong, but its heuristic landscape is clearly not the
same as Hex's.

This last point is perhaps the most important engineering lesson of the
project. The Y branch did not need a completely new software design. It needed
the Hex architecture to be abstract enough to accommodate a different win
condition while remaining concrete enough to stay fast. Once that backbone was
in place, the remaining challenge was no longer engineering parity but
game-specific knowledge.

\section{Conclusion}

This repository now contains a coherent study of Monte Carlo Tree Search across
Hex and Y. The classical Hex branch reproduces the main qualitative findings of
\emph{Monte-Carlo Hex}. The learned Hex branch shows that Gumbel AlphaZero can
outperform a classical baseline despite using far fewer simulations per move.
Most importantly for the present extension, the Y branch is no longer a draft.
It now has correct rule handling, a Python/Cython structure matching the Hex
baseline, regression coverage, saved experiment outputs, and a fast backend
suitable for full-size sweeps.

The full Y experiments also change the interpretation of the module. The branch
does not merely pass sanity checks; it already shows the expected simulation
scaling, a clear preference for $C_{\mathrm{uct}}=0$ under RAVE, and a
substantial gap between random and connectivity-aware playouts. The next step
is therefore not more implementation cleanup but better Y-specific knowledge.
The infrastructure is in place. What remains open is which structural
heuristics or priors will do for Y what bridges do for Hex.
